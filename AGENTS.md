# AGENTS.md

本文件是 `TMM-interference-spectrum` 项目的全局开发规范。所有参与本项目的 AI Agent，包括但不限于 Codex、Claude、Copilot 及其他自动化开发助手，在开始任何任务前必须优先阅读、理解并严格遵守本文件。

如果本文件与临时对话指令冲突，以本文件中的项目工程规范、物理建模约束、目录隔离规则和上下文同步规则为优先；如确有必要偏离，必须先明确说明原因、影响范围和替代方案。

## 1. Project Overview & Phase Tracking

### 1.1 项目目标
本项目旨在基于严密传递矩阵法（Transfer Matrix Method, TMM）与 Levenberg-Marquardt（LM）优化算法，开发一套用于非接触式定量解析钙钛矿太阳能电池（Perovskite Solar Cells, PSCs）内部 Z 轴光学结构的干涉反演引擎。

项目的目标不是单纯“拟合一条曲线”，而是建立一条从实验光谱输入、物理前向建模、参数反演、结果验证到输出解释的可复现计算链路。任何代码修改都必须服务于以下核心目标：

- 保证光学模型的物理一致性
- 保证反演流程的数值稳定性
- 保证数据处理链条的可追溯性
- 保证每个阶段结果可复现、可比较、可回滚

### 1.2 项目当前工作方式
项目采用“Phase 连续编号”的方式进行开发管理、结果归档和 Git 提交标记。每一项重要开发任务、每一个关键能力上线、每一轮可验证的实验改进，都必须显式归属于某个 Phase。

示例：
- `Phase 01: 绝对反射率标定`
- `Phase 02: TMM 基准前向仿真`
- `Phase 03: 初始 LM 反演闭环`
- `Phase 04: 多层参数敏感性验证`

### 1.3 Agent 的 Phase 责任
Agent 在执行任何任务前，必须先做以下判断：

1. 当前任务属于哪个 Phase
2. 当前任务是：
   - 新 Phase 的启动
   - 既有 Phase 的补丁修复
   - 跨 Phase 的结构性重构
3. 本次修改会影响哪些输入、输出、参数定义、结果解释

若用户未明确指定 Phase，Agent 必须先根据上下文判断当前最合理的 Phase，并在工作说明、提交信息、文档同步中保持一致命名。

### 1.4 Phase 标识要求
以下位置必须显式带上当前 Phase：
- Git commit message
- `docs/PROJECT_STATE.md` 的更新记录
- 新增或更新的 SOP 文档
- 重要结果图或结果说明中的标题/注释
- 涉及阶段性产物的结果目录说明

## 2. Strict Directory & File Structure

### 2.1 根目录红线
严禁将以下内容直接放在项目根目录：
- Python 脚本
- 临时测试脚本
- CSV 数据
- 图表文件
- `.mat`/`.npy`/`.npz` 等中间结果
- 调试输出日志
- 临时笔记草稿

项目根目录只允许保留项目级元信息与入口性文件，例如：
- `README.md`
- `AGENTS.md`
- `.gitignore`
- 必要的环境配置文件
- 项目级构建/依赖文件

如发现历史遗留文件放置错误，Agent 不得忽略，必须在合适时机提出整理建议，或在本次任务明确范围内完成迁移并同步文档。

### 2.2 标准目录架构
项目必须维持如下结构，Agent 在新增文件时必须严格遵守：

```text
TMM-interference-spectrum/
├── AGENTS.md
├── README.md
├── src/
│   ├── core/
│   └── scripts/
├── data/
│   ├── raw/
│   └── processed/
├── resources/
├── results/
│   ├── figures/
│   └── logs/
└── docs/
```

### 2.3 各目录职责定义

#### `src/`
存放所有核心算法代码，不允许混入原始数据、结果图和实验输出。

#### `src/scripts/`
用于存放可独立执行的流水线脚本。命名建议采用顺序化风格，便于表达处理阶段与依赖关系，例如：
- `step01_calibrate_reflectance.py`
- `step02_build_tmm_forward_model.py`
- `step03_run_lm_inversion.py`

此目录下的脚本应具备以下特征：
- 明确输入路径
- 明确输出路径
- 可独立运行
- 不依赖对话上下文
- 不通过隐式全局变量串联前后步骤

#### `src/core/`
用于存放可复用的核心物理函数、数值模块、数据模型和公共工具。任何跨脚本复用超过一次的逻辑，应优先考虑下沉到本目录，而不是复制粘贴到多个脚本中。

适合放入 `src/core/` 的内容包括：
- TMM 相关矩阵计算函数
- 折射率/消光系数插值逻辑
- 角度、波长、厚度单位转换
- 残差函数与目标函数定义
- LM 优化封装
- 数据校验函数
- 文件 I/O 适配器

#### `data/raw/`
存放原始实验数据。该目录视为只读区。
Agent 不得覆盖、重写、重命名或就地清洗 `data/raw/` 中的数据文件，除非用户明确要求并说明依据。

#### `data/processed/`
存放由脚本生成的标准中间数据。所有跨脚本传递的数据必须优先通过此目录完成，而不是通过复制代码逻辑或保存在脚本内存状态中。

典型文件包括：
- 标定后的绝对反射率 CSV
- 统一波长轴后的数据
- 反演输入模板
- 参数初值表
- 中间拟合结果表

#### `resources/`
存放外部物理常量库、材料色散数据库、参考文献摘录、实验标定资源等。

典型内容包括：
- ITO 光学常数 `.mat`
- 理论色散模型说明文档
- 参考材料的 `n-k` 数据
- 光学常量换算说明

该目录中的内容应视为“资源依赖”，不是运行输出目录。

#### `results/`
存放脚本运行产生的输出结果，禁止将结果散落到其他目录。

建议子目录至少包括：

##### `results/figures/`
存放图表输出：
- `.png`
- `.pdf`
- 对比图
- 拟合结果图
- 参数敏感性曲线

##### `results/logs/`
存放运行日志、实验记录摘要、残差统计、批量运行信息等文本输出。

如项目后续需要，也可新增：
- `results/tables/`
- `results/checkpoints/`
- `results/reports/`

但新增前必须保证命名含义明确，并同步更新 `docs/PROJECT_STATE.md`。

#### `docs/`
存放项目文档和上下文同步文件。所有 Agent 必须把这里视为“可持续上下文层”，而不是临时草稿区。

## 3. Context Synchronization Protocol

### 3.1 核心原则
为了防止长对话、多人协作、跨阶段开发导致项目上下文丢失，Agent 必须主动维护 `docs/PROJECT_STATE.md`。该文件是架构师和后续 Agent 快速理解“系统当前长什么样”的唯一可信摘要入口。

### 3.2 强制维护时机
出现以下任一情况时，必须更新 `docs/PROJECT_STATE.md`：

- 完成一个重要功能模块
- 完成一个新的 Phase
- 修改了核心目录结构
- 修改了关键脚本间的数据流
- 引入了新的外部资源依赖
- 修复了影响主流程的重大 Bug
- 变更了参数定义、单位约定或结果解释方式

### 3.3 `PROJECT_STATE.md` 必须包含的内容

#### A. 最新目录树
必须包含精简但准确的目录树，至少覆盖：
- `src/`
- `data/`
- `resources/`
- `results/`
- `docs/`

要求：
- 只展示关键层级
- 不堆砌无关文件
- 保持与真实仓库一致

#### B. 核心脚本数据流图（SOP）
必须说明关键脚本之间的输入输出关系。建议采用如下格式：

```md
step01_xxx.py
- 输入：data/raw/xxx.csv
- 输出：data/processed/yyy.csv
- 作用：完成绝对反射率标定

step02_xxx.py
- 输入：data/processed/yyy.csv, resources/ito.mat
- 输出：results/figures/baseline_fit.png
- 作用：执行 TMM 前向模拟并生成基线结果
```

如流程复杂，可补充 Mermaid 图，但必须先保证文字版可直接阅读。

#### C. Bug 挂起区
必须记录仍未解决的问题，包括：
- Bug 表现
- 可能根因
- 当前影响范围
- 暂未解决原因
- 后续建议排查方向

严禁只写“有问题待解决”这种空泛描述。

#### D. 架构风险区
必须记录潜在风险，例如：
- 单位系统尚未统一
- 某模块耦合过深
- 某类材料参数来源不稳定
- 优化初值敏感性过高
- 某脚本承担了过多职责

#### E. 最近一次更新摘要
必须记录：
- 更新时间
- 当前 Phase
- 本次新增/修改了哪些核心能力
- 哪些结论已验证
- 哪些结论仍待验证

### 3.4 文档风格要求
`PROJECT_STATE.md` 必须满足以下标准：
- 精简
- 准确
- 面向接手者
- 能让不了解最近对话的人快速恢复项目上下文
- 不写空话
- 不复制大段源码
- 不写无验证依据的推断结论

## 4. Git Version Control & Commit Protocol

### 4.1 提交原则
每次成功运行并通过验证的修改，Agent 必须尽快提交 Git，避免在多个逻辑步骤之间积累过大的未提交状态，确保每一轮可工作的结果都可回滚、可对比、可审计。

### 4.2 标准提交流程
如果本次任务已完成“实现 + 基本验证 + 结果确认”，应执行以下流程：

```bash
git add .
git commit -m "[Phase XX] 中文描述"
git push
```

若当前仓库没有远程仓库，或用户尚未配置 push 权限，可跳过 `git push`，但必须明确说明原因。

### 4.3 Commit Message 强制规范
Commit message 必须满足以下要求：
- 必须使用中文
- 必须以 `[Phase XX]` 开头
- 必须说明修改主题
- 必须尽量体现解决的问题，而不是只写“更新代码”

示例：
- `[Phase 02] 重构TMM前向仿真脚本并统一波长轴插值逻辑`
- `[Phase 03] 修复LM残差函数维度错误并补充反演输出CSV`
- `[Phase 04] 新增ITO色散资源加载模块并修正反射率标定流程`

### 4.4 提交前最小检查
提交前至少确认以下内容：
- 代码能运行
- 输入输出路径正确
- 没有把临时文件误放在根目录
- 结果文件落在正确目录
- 文档同步已完成
- 未引入明显未解释的物理假设变化

### 4.5 禁止行为
严禁以下行为：
- 未验证就提交
- 把多个无关任务混成一个 commit
- 用英文或无意义信息提交
- 提交临时调试垃圾文件
- 通过回避问题的方式制造“表面成功”的提交

## 5. Communication & Coding Protocols

### 5.1 增量修改原则（Delta Protocol）
在对话沟通或开发说明中，Agent 严禁无必要地粘贴几百行完整旧代码。应采用以下格式：
- 说明修改目标
- 指出修改文件
- 给出关键修改片段
- 用 Anchor 说明修改位置
- 解释为何修改

推荐格式：
```md
文件：src/core/tmm_solver.py
定位：`compute_reflectance()` 中构造界面矩阵的位置
修改：将 s/p 偏振分支统一抽象为单独函数，减少重复并修复角度传播项符号
原因：此前 p 偏振公式中的分母定义与推导不一致
```

### 5.2 Traceback 优先原则
任何异常发生后，Agent 必须先完整阅读 Traceback，并先判断错误属于哪一类：

#### A. 数学物理错误
例如：
- 对数域出现非正值
- 矩阵病态或奇异
- 复数平方根支路选取错误
- 角度传播导致非物理结果
- 单位不一致导致数量级爆炸
- 插值超出材料数据库有效范围

#### B. 代码逻辑错误
例如：
- 数组维度不一致
- 路径错误
- CSV 列名不匹配
- 参数顺序写错
- 返回值类型错误
- 未初始化变量

Agent 必须明确写出判断依据，不能把所有错误都归因于“代码 bug”。

### 5.3 Crash Ledger 规则
当出现异常时，必须记录：
- 错误位置
- 错误类型
- 直接触发条件
- 可能根因
- 属于物理问题还是逻辑问题
- 当前采用的修复策略

严禁以下行为：
- 用 `try-except` 吞掉报错继续运行
- 用默认值伪装修复
- 无依据裁剪公式边界来掩盖发散
- 擅自截断积分、相位项或损耗项以换取“可运行”
- 把 `NaN`/`Inf` 静默替换成 0 而不说明

### 5.4 数据解耦法则
脚本之间必须通过 `data/processed/` 下的标准文件进行数据传递。禁止以下做法：
- 在新脚本中复制前置脚本的长处理逻辑
- 在多个脚本中维护各自不同版本的数据清洗代码
- 依赖对话中提到的中间变量作为事实来源
- 把实验流程写成只能在当前会话中成立的“一次性脚本链”

所有跨步骤共享数据，都必须落盘为标准中间文件，并在文档中注明来源与用途。

### 5.5 单一职责原则
每个脚本、模块、函数都应尽量职责单一。
如果一个脚本同时负责：
- 读取原始数据
- 完成复杂清洗
- 构建物理模型
- 执行优化
- 画图
- 导出报告

则应视为高风险设计，应拆分。

### 5.6 命名与可读性要求
命名必须体现物理含义和工程角色，避免：
- `tmp`
- `test2`
- `new_final`
- `fix_ok`
- `data1`

推荐命名：
- `wavelength_nm`
- `reflectance_abs`
- `layer_thickness_nm`
- `lm_residual_vector`
- `interpolated_nk_table`

### 5.7 注释规范
注释必须解释“为什么”，而不是重复“做了什么”。
优先给以下内容补注释：
- 公式实现来源
- 单位约定
- 特殊边界条件
- 数值稳定性处理
- 某段实现与论文定义的映射关系

## 6. Physics & Numerical Modeling Rules

### 6.1 物理正确性优先
对于本项目，物理正确性优先级高于代码表面整洁、高于抽象优雅、高于局部性能优化。
任何会改变物理意义的改动，必须先说明：
- 改动前假设
- 改动后假设
- 对结果的潜在影响
- 如何验证其合理性

### 6.2 单位系统必须显式
所有核心变量必须明确单位，尤其是：
- 波长
- 厚度
- 折射率实部/虚部
- 吸收系数
- 角度
- 光谱采样步长

禁止在同一流程中隐式混用：
- `nm` 与 `um`
- `degree` 与 `radian`
- `frequency` 与 `wavelength`

如确需转换，必须在代码和文档中显式标明。

### 6.3 公式实现必须可追溯
任何关键公式实现都应能回答以下问题：
- 来源于哪类理论或文献
- 对应代码在哪个函数
- 输入输出变量分别对应什么物理量
- 是否存在数值近似
- 适用边界条件是什么

### 6.4 稳定性处理必须有物理与数值依据
可以做稳定性处理，但必须满足：
- 有明确理由
- 不改变核心物理意义
- 可被解释
- 可被回滚
- 已记录到文档或代码注释

不允许用“为了避免报错”作为唯一理由去改变模型。

### 6.5 优化与反演规则
涉及 LM 或其他优化器时，必须注意：
- 参数初值来源是否合理
- 参数边界是否具有物理意义
- 残差定义是否与观测量一致
- 优化失败是否有可解释诊断
- 结果是否存在多解性或病态敏感性

反演成功不等于物理可信。必须区分：
- 数值上收敛
- 物理上合理

## 7. Script Pipeline & I/O Contracts

### 7.1 每个脚本都必须具备明确 I/O 合约
每个 `src/scripts/` 下的脚本，都必须能回答以下问题：
- 输入文件路径是什么
- 输出文件路径是什么
- 依赖哪些资源文件
- 依赖哪些前序脚本产物
- 成功运行后会生成哪些结果

### 7.2 输入输出路径必须稳定
禁止在代码中硬编码临时桌面路径、下载目录路径或用户私有路径。
所有路径必须相对项目根组织，或由清晰参数传入。

### 7.3 输出命名要求
输出文件命名应尽量体现：
- 当前 Phase
- 步骤编号
- 数据类型
- 版本或日期

示例：
- `phase02_calibrated_reflectance.csv`
- `phase03_lm_fit_result.csv`
- `phase03_forward_vs_measured.png`

### 7.4 图表输出要求
所有图表必须放在 `results/figures/`，并满足：
- 文件名可读
- 坐标轴有单位
- 图例准确
- 如涉及拟合，应能区分实验数据与模型曲线
- 如涉及阶段对比，应标明 Phase

- 相关目录是否符合规范
- 是否需要先阅读 `docs/PROJECT_STATE.md`
- 本次修改会不会影响数据流、目录结构或物理假设
- 是否需要同步更新文档
- 是否需要准备 Git 提交

Agent 在完成任务后必须检查：

- 代码是否运行
- 结果是否合理
- 输出目录是否正确
- 是否产生了不该在根目录出现的文件
- `docs/PROJECT_STATE.md` 是否需要更新
- 是否满足提交条件
- commit message 是否符合 `[Phase XX] 中文描述` 规范

## 10. Forbidden Behaviors

以下行为在本项目中视为明确违规：

- 将 Python 脚本、CSV、图表直接丢到项目根目录
- 不区分物理错误和逻辑错误，笼统修 bug
- 为了“跑通”而吞异常、裁公式、压 NaN
- 在脚本间复制粘贴冗长处理逻辑
- 在没有验证的情况下声称“修复完成”
- 修改了核心流程却不更新 `docs/PROJECT_STATE.md`
- 不按 Phase 管理结果和提交
- 使用含糊不清的命名和无意义 commit message
- 把一次性对话上下文当作长期项目记忆
- 让中间数据只存在于内存或聊天记录中而不落盘

---

## 11. Final Operating Principle

本项目不是通用 CRUD 工程，而是一个以物理建模正确性、数值稳定性、数据链路可追溯性为核心的科研型光学计算与反演系统。

所有 Agent 必须遵循以下最高原则：

1. 先理解物理目标，再动代码
2. 先厘清数据流，再拼接脚本
3. 先读 Traceback，再判断修复策略
4. 先保证目录与文档整洁，再扩展功能
5. 先完成最小可验证闭环，再推进下一 Phase

若无法同时满足“快速推进”和“物理严谨”，优先选择物理严谨。
若无法同时满足“局部方便”和“长期可维护”，优先选择长期可维护。

本文件自生效起，视为本项目所有 AI Agent 的强制系统级开发规范。

### 6. Literature-First Policy (文献驱动与 RAG 准则)

#### 6.1 文献存储与索引

- 所有外部参考文献，包括原始 PDF、MinerU 解析后的 Markdown 文件夹及其附属资源，固定存放在 `resources/references/` 目录。
- 项目的核心文献索引地图固定为 `docs/LITERATURE_MAP.md`。
- `docs/LITERATURE_MAP.md` 是本项目关于外部物理知识、公式出处、参数来源和证据链组织的长期记忆入口。
- 若现有文献仍散落在其他历史目录中，后续 Agent 应以迁移到 `resources/references/` 为目标，而不是继续扩散新的存放位置。

#### 6.2 代码实现强制约束（Literature-First）

- 在实现任何涉及物理模型、数学推导、数值方法、核心参数选型的代码前，必须且绝对必须先阅读 `docs/LITERATURE_MAP.md`。
- 上述“核心参数选型”包括但不限于：
  - 色散模型参数
  - 厚度先验范围
  - 边界条件选取
  - 材料常数
  - 拟合窗口
  - 优化初值和约束
- 确定最相关的文献条目后，必须回源打开 `resources/references/` 下对应的详细 Markdown 原文，逐项确认公式、常数、单位、适用条件和符号定义。
- 严禁在缺少文献依据时凭空捏造物理定律、单位换算、经验参数或论文结论。
- 若文献之间存在冲突，必须显式记录冲突来源、采用理由和影响范围，不得静默挑选“更方便实现”的版本。
- 在涉及复杂物理逻辑的代码、函数注释或 docstring 中，必须以注释形式标注所依据的文献条目 ID，例如 `[LIT-0001]`。
- 若一个实现同时依赖多篇文献，应并列标注多个条目 ID，并确保 `docs/LITERATURE_MAP.md` 能追溯到对应原文位置。

#### 6.3 文献地图更新触发机制

- 如果发现所需的物理依据、公式出处、参数来源或实验约束未被 `docs/LITERATURE_MAP.md` 收录，Agent 必须先补全文献地图，再继续实现相关代码。
- 如果发现 `resources/references/` 下新增了文献，Agent 必须主动更新 `docs/LITERATURE_MAP.md`，确保新增文献可被后续任务检索和引用。
- 如果发现文献目录结构、文献命名或 MinerU 解析结果发生变化，Agent 必须同步修正文献地图中的路径和索引信息。
- 遇到上述任一情况时，Agent 必须主动调用项目内置的 `$literature-map` 技能（Skill）去扫描并更新文献地图。
- 必须确保 `docs/LITERATURE_MAP.md` 永远处于最新状态；任何依赖外部文献的开发任务，都不得建立在过期的文献地图之上。

#### 6.4 RAG 使用原则

- `docs/LITERATURE_MAP.md` 负责回答“应该先看哪篇文献、哪一节、哪一类公式”。
- `resources/references/` 下的详细 Markdown 原文负责回答“具体公式是什么、参数是什么、单位是什么、适用条件是什么”。
- Agent 应先索引、再回源、后实现；不得跳过回源确认步骤直接编码。
- 文献驱动优先级高于经验猜测、高于习惯写法、高于未经验证的二手总结。
